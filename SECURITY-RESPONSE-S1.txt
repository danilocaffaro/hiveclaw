C1 (Auth bypass) — Sherlock diz P0, eu digo P1

O HiveClaw foi desenhado para single-user localhost (como Ollama, LocalAI, LM Studio). O "auth bypass" não é bypass — é design constraint. Quando o SPA faz fetch para localhost:4070, não tem header de API key porque é same-origin.

O risco real: existe quando alguém faz port-forward sem entender. Mas isso é equivalente a expor Ollama na internet sem proxy reverso — é responsabilidade do operador.

Mitigação já presente: CORS impede cross-origin (site malicioso não consegue fazer fetch para localhost:4070 do browser da vítima).

Quando vira P0: se mudarmos para multi-user ou cloud deploy. Aí sim, JWT/session auth obrigatório. Não é o caso do v1.0.

---

C2 (Bash blocklist) — Sherlock diz P1, eu digo icebox

Concordo 100% que blocklist é security theater. Os bypasses listados são todos válidos.

Mas a abordagem do HiveClaw não é SÓ blocklist — é Operational Awareness. O system prompt recebe contexto runtime (OS, cwd, port, platform) + regras de self-preservation. O agente entende que rm -rf / destrói seu próprio ambiente. Isso é mais robusto que qualquer blocklist.

A alternativa correta (firejail/bubblewrap) é Linux-only e requer mudanças significativas na arquitetura de tools. Para v1.0 self-hosted com usuário técnico, Operational Awareness é suficiente. Para multi-tenant SaaS, sandbox obrigatório — mas isso é v2.0+.

---

C5 (Plaintext creds) — Sherlock diz P1, eu digo P1 mas com caveat

Encrypt-at-rest sem auth layer não resolve nada — se o atacante tem acesso ao server, ele pega a chave junto com o DB. C5 só faz sentido depois de C1 (auth). Sequenciar como C1→C5 para v1.1.
